# Liquity API Rate Limits

The Liquity API (api.liquity.org) is served as static files via GitHub Pages.
There are no formally documented rate limits imposed by Liquity on consumers.

## Effective Limits

| Resource | Limit | Notes |
|----------|-------|-------|
| GitHub Pages requests | ~unlimited (CDN) | Subject to GitHub's standard abuse policies |
| Data freshness | ~hourly (cron) | Files are regenerated by scheduled GitHub Actions |
| Historical data | N/A | Only latest snapshot is served; no time-series query capability |

## Data Update Cadence

| Endpoint Group | Update Frequency |
|----------------|-----------------|
| V1 supply files (lusd_total_supply, lqty_circulating_supply) | Hourly via GitHub Actions |
| V2 protocol stats (ethereum.json, mainnet.json) | Hourly via GitHub Actions |
| V2 website data (bold-venues, leaderboard, borrow-rates) | Hourly via GitHub Actions |
| Governance epochs (latest_completed_epoch, initiatives) | Weekly (epoch = 604,800s) |

## RPC / SDK Usage

Developers using the `@liquity/lib-ethers` SDK directly must supply their own
Ethereum RPC endpoint (Alchemy, Infura, etc.) and are subject to the rate
limits of their chosen RPC provider, not Liquity.

## Recommendations

- Cache responses locally; files change at most hourly.
- Do not poll more frequently than every 5 minutes — data will not be fresher.
- For real-time on-chain data, use the Liquity SDK with a personal RPC key.
- For historical analytics, use The Graph's Liquity subgraph or Dune Analytics
  (Liquity publishes several public Dune dashboards).
